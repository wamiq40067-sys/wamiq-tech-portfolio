CISSP Security Domains – Simplified Notes
Domain 1: Security and Risk Management

This domain focuses on building and maintaining an organization’s overall security posture, which is its ability to protect systems, assets, and data while adapting to evolving threats.

Key areas include:

Security goals and policies
Risk management and mitigation
Regulatory compliance
Business continuity planning
Legal and ethical responsibilities

Information Security (InfoSec) is a major part of this domain and involves processes designed to protect organizational data and systems.

Common InfoSec activities include:

Incident response
Vulnerability management
Application security
Cloud security
Infrastructure protection

Organizations may also use playbooks and employee training programs to improve security awareness and incident response readiness.

For example, companies handling personal data may need to update security procedures to comply with regulations such as the General Data Protection Regulation (GDPR).

Domain 2: Asset Security

Asset security focuses on protecting an organization’s physical and digital assets throughout their lifecycle.

This includes:

Data storage
Maintenance
Retention
Backup management
Secure destruction of data

Because stolen or exposed assets can increase organizational risk, proper asset tracking and recovery planning are essential.

Security teams often create backups and disaster recovery strategies to ensure data can be restored after security incidents.

Domain 3: Security Architecture and Engineering

This domain focuses on designing secure systems, tools, and processes that protect organizational data and infrastructure.

Security architects and engineers implement security models and design principles to reduce risks.

Important concepts include:

Threat modeling
Least privilege
Defense in depth
Zero trust
Separation of duties
Fail securely
Simplicity in design

A shared responsibility approach is also important, meaning everyone involved in system design contributes to improving security.

For example, organizations may use SIEM tools to detect suspicious login activity and identify possible unauthorized access attempts.

Domain 4: Communication and Network Security

This domain focuses on securing network infrastructure and communication systems, including:

Physical networks
Wireless communications
Remote access systems
Cloud-based communication environments

Organizations with hybrid or remote work environments must ensure employees can securely access company resources from different locations.

Security controls such as restricted access, VPNs, and network monitoring help protect organizational networks.

Domain 5: Identity and Access Management (IAM)

The IAM domain focuses on verifying user identities and controlling access to systems and data.

Its goal is to:

Prevent unauthorized access
Ensure users only access what they need

This domain commonly uses the principle of least privilege, which means users receive only the minimum permissions required to perform their tasks.

For example, customer support employees may only have temporary access to limited customer information while resolving an issue.

Domain 6: Security Assessment and Testing

This domain involves evaluating systems for risks, vulnerabilities, and security weaknesses.

Common activities include:

Security assessments
Penetration testing
Security audits
Control testing
Data analysis

Organizations use these practices to identify weaknesses before attackers can exploit them.

Cybersecurity professionals may also review user permissions and system configurations to ensure security policies are being followed.

Domain 7: Security Operations

This domain focuses on monitoring, detecting, responding to, and recovering from security incidents.

Key operational activities include:

Incident response
Threat monitoring
Intrusion detection and prevention
Log management
SIEM monitoring
Reporting and documentation
Digital forensics
Security awareness training

Security operations teams work together to investigate suspicious activity and reduce the impact of cyberattacks.

After incidents occur, teams review lessons learned to improve future defenses.

Domain 8: Software Development Security

This domain focuses on integrating security into the software development lifecycle (SDLC).

The goal is to build secure applications by applying secure coding practices during:

Design
Development
Testing
Deployment
Maintenance

Security should be considered throughout the entire development process rather than being added later.

Common practices include:

Application security testing
Code reviews
Vulnerability scanning
Encryption implementation
Quality assurance testing

For example, security analysts may verify that sensitive user or patient data is properly encrypted within software systems or devices.

Risk Management Strategies

Organizations use different approaches to manage cybersecurity risks:

Acceptance

Accepting a risk when avoiding or reducing it may disrupt business operations.

Avoidance

Eliminating activities or situations that create the risk.

Transference

Shifting responsibility for the risk to a third party, such as an insurance provider or vendor.

Mitigation

Reducing the likelihood or impact of a known risk through security measures.

Security Frameworks

Organizations often follow industry-recognized frameworks to strengthen risk management and protect assets.

Examples include:

National Institute of Standards and Technology Risk Management Framework (NIST RMF)
Health Information Trust Alliance (HITRUST)

These frameworks provide structured guidance for managing cybersecurity threats, risks, and vulnerabilities.

Common Threats, Risks, and Vulnerabilities
Threats

A threat is any event or circumstance capable of harming systems, data, or organizational assets.

Common Threat Types
Insider Threats

Employees, contractors, or vendors misuse authorized access to damage the organization or steal information.

Advanced Persistent Threats (APTs)

Highly skilled attackers maintain unauthorized access to systems over long periods without detection.

Risks

A risk is anything that can negatively affect the confidentiality, integrity, or availability of an asset.

Risk is often evaluated based on the likelihood of a threat occurring and the potential impact it could cause.

Common Risk Factors
External Risks

Threats originating outside the organization, such as cyberattacks or unauthorized access attempts.

Internal Risks

Risks caused by employees, former staff members, vendors, or trusted partners.

Legacy Systems

Outdated systems or devices that are no longer properly maintained or secured.

Multiparty Risks

Risks introduced when third-party vendors gain access to sensitive systems or intellectual property.

Software Compliance and Licensing Risks

Outdated software, missing patches, or non-compliant systems that increase exposure to attacks.

Industry Resources

Cybersecurity organizations regularly publish guidance on current risks and vulnerabilities.

Examples include:

National Institute of Standards and Technology (NIST)
Open Web Application Security Project (OWASP)

OWASP publishes the “Top 10 Web Application Security Risks,” which highlights major web security threats and is updated periodically.

Recent additions include:

Insecure design
Software and data integrity failures
Server-side request forgery (SSRF)

These updates demonstrate how cybersecurity threats constantly evolve over time.

Vulnerabilities

A vulnerability is a weakness in a system that attackers can exploit.

Organizations must continuously monitor and patch vulnerabilities to reduce security risks.

Examples of Common Vulnerabilities
ProxyLogon

A Microsoft Exchange Server vulnerability that allows remote attackers to deploy malicious code.

ZeroLogon

A flaw in Microsoft’s Netlogon authentication protocol that can allow attackers to gain unauthorized access.

Log4Shell

A major Java-based vulnerability that enables remote code execution and data exposure.

PetitPotam

A Windows NTLM-related attack technique that can trigger unauthorized authentication requests.

Security Logging and Monitoring Failures

Weak monitoring systems that allow attackers to operate without detection.

Server-Side Request Forgery (SSRF)

A vulnerability that manipulates servers into accessing internal or backend resources.

Vulnerability Management

Cybersecurity analysts often work in vulnerability management, which involves:

Identifying weaknesses
Monitoring systems continuously
Applying updates and patches
Reducing organizational exposure to threats

Regular patching and monitoring are essential because even known vulnerabilities remain dangerous if they are not properly fixed.

CIA triad 
A model that helps inform how organizations consider risk when setting up systems and security policies.

confidentiality - Only authorized users can access specific assets or data.

Integrity - The data is correct, authentic, and reliable.

Availability - Data is accessible to those who are authorized to access it.




The CIA triad is a fundamental cybersecurity model used by organizations to evaluate risks and build secure systems and policies. It consists of three key principles: confidentiality, integrity, and availability. These principles help organizations maintain a strong security posture, which refers to their ability to protect important systems and data while responding effectively to security challenges and changes.

Confidentiality

Confidentiality ensures that sensitive information is only accessible to authorized individuals. Organizations often apply the principle of least privilege, which gives users access only to the resources necessary for their job responsibilities. Restricting unnecessary access helps protect private and confidential data from unauthorized use or exposure.

Integrity

Integrity focuses on ensuring that data remains accurate, authentic, and trustworthy. Organizations use different methods to maintain data integrity, including cryptography and encryption. Cryptography helps secure information by preventing unauthorized users from altering or reading it, while encryption converts readable data into encoded information to reduce the risk of tampering or unauthorized access.

Availability

Availability means that authorized users can access systems and data whenever needed. Organizations maintain availability by ensuring reliable access to networks, applications, and resources, including for remote employees. Access is still controlled based on job roles, so employees can only use the information necessary for their specific duties. For example, accounting staff may access financial records but not confidential development data.

Establish Secure Defaults

This principle states that an application should be secure by default. Users should automatically operate in the safest configuration, and making the system less secure should require additional actions or changes.

Fail Securely

Fail securely means that if a security control or system stops working, it should switch to the safest possible state. For example, if a firewall encounters a failure, it should block all traffic and close connections instead of allowing unrestricted access.

Don’t Trust Services

Organizations often rely on third-party vendors and external services, but those partners may follow different security standards. Because of this, organizations should not automatically assume external systems are secure. For instance, if an airline uses a third-party service to manage customer reward points, the airline should verify the information before presenting it to customers.

Avoid Security by Obscurity

A system should not depend on secrecy alone for protection. Hiding source code or system details is not enough to guarantee security. Strong protection should instead come from multiple security measures such as secure passwords, layered defenses, reliable network design, monitoring systems, and audit controls.

Security Audits

A security audit is an independent evaluation of an organization’s security policies, procedures, and controls to determine whether they meet required standards and expectations. These expectations may include internal company policies, industry best practices, legal requirements, and government regulations.

Security audits are also used to examine the effectiveness of security controls, which are safeguards implemented to reduce risks and protect systems and data. Regular audits help organizations verify that security activities, such as monitoring security dashboards and identifying threats or vulnerabilities, are being carried out properly. If weaknesses are discovered, organizations must establish a remediation process to resolve them and maintain a strong security posture.

Goals and Objectives of a Security Audit

The main goal of a security audit is to ensure that an organization’s information technology practices align with organizational standards and industry requirements. Audits help identify weaknesses, areas for improvement, and security gaps that need corrective action.

Another important objective is to protect sensitive data and avoid penalties or fines related to noncompliance with laws and regulations. The frequency of audits often depends on government requirements, industry regulations, and organizational policies.

Factors That Influence Audits

Several factors determine the type and frequency of audits an organization may conduct, including:

The industry the organization operates in
The size of the organization
Government regulations that apply to the business
The organization’s geographical location
Voluntary compliance with specific regulatory standards
Role of Frameworks and Controls in Audits

Security frameworks and controls play an important role in preparing organizations for audits. Frameworks such as the National Institute of Standards and Technology Cybersecurity Framework (NIST CSF) and the International Organization for Standardization 27000 series provide structured guidance that helps organizations meet compliance requirements more effectively.

When combined with security controls, these frameworks improve an organization’s ability to meet security standards and simplify both internal and external audits.

The three primary categories of security controls reviewed during audits are:

Administrative or managerial controls
Technical controls
Physical controls
Audit Checklist

Before starting an audit, organizations typically create an audit checklist that outlines the audit process and objectives.

Identify the Scope of the Audit

The audit scope should define:

Which assets and systems will be evaluated
The goals of the audit
How frequently audits should occur
Whether policies, procedures, and protocols are functioning properly and being followed by employees
Perform a Risk Assessment

A risk assessment helps identify and evaluate risks related to internal operations, budgets, security controls, and compliance requirements.

Conduct the Audit

During the audit process, the organization examines the security of the identified systems, assets, and controls to determine whether they meet required standards.

Develop a Mitigation Plan

A mitigation plan is created to reduce identified risks and minimize the likelihood of financial losses, penalties, or security incidents that could impact the organization.

Share Results With Stakeholders

After the audit is completed, a detailed report is provided to stakeholders. This report includes findings, identified risks, recommended improvements, and any compliance standards the organization must continue to follow.

Current SIEM Solutions
A Security Information and Event Management (SIEM) system is a software solution that gathers and examines log data to track important activities within an organization. These tools provide real-time monitoring of security events and help security teams detect possible threats, risks, or vulnerabilities. The collected data is analyzed to support investigations and improve overall security management.

SIEM platforms also include different types of dashboards that allow cybersecurity professionals to organize, monitor, and manage large amounts of security-related information. Despite their advanced capabilities, most existing SIEM solutions still depend heavily on human analysts to review alerts and interpret security events before action can be taken.

The Future of SIEM Tools
As cybersecurity technologies continue to advance, organizations increasingly rely on cloud-based solutions. Modern SIEM platforms are evolving to support both cloud-hosted and cloud-native environments.

Cloud-hosted SIEM solutions are maintained by third-party vendors who manage the infrastructure, updates, and maintenance. Organizations can access these tools through the internet without building their own systems, making them cost-effective and easier to manage.

Cloud-native SIEM solutions go a step further by being specifically designed for cloud environments. They make better use of cloud computing features such as scalability, flexibility, and high availability, allowing organizations to process and store larger volumes of data efficiently.

The future development of SIEM technology is also being influenced by emerging technologies such as the Internet of Things (IoT). As more internet-connected devices are introduced, the number of potential attack points increases, generating even more security data that must be analyzed. This growth will require SIEM systems to become more intelligent and capable of handling diverse and complex threats.
In addition, advancements in Artificial Intelligence (AI) and Machine Learning (ML) are expected to improve SIEM capabilities significantly. These technologies can enhance threat detection, automate pattern recognition, improve data visualization, and optimize data storage processes.

Automation is also becoming an important feature in cybersecurity operations. Security Orchestration, Automation, and Response (SOAR) solutions are being integrated with SIEM platforms to automate responses to common security incidents. By reducing the need for manual intervention, SOAR allows security analysts to focus on more advanced and unusual threats that cannot easily be automated.

Overall, future cybersecurity platforms are expected to become more interconnected, enabling systems and devices to communicate and coordinate security responses more effectively. Although this level of integration is still developing, it represents an important direction for the future of cybersecurity operations.

Open-Source Tools

Open-source tools are software solutions that are usually available at no cost and are often designed to be accessible and easy to use. These tools are developed collaboratively by communities of developers and contributors, which can improve both innovation and security. Because the source code is publicly available, users are able to customize the software according to their own requirements, leading to the creation of many different services and features from the same base software.

Developers create open-source projects with the goal of improving software quality while allowing others to use, study, and modify the code under specific licensing terms. In addition to the source code, training resources and documentation are commonly provided, making it easier for users to learn, adapt, and enhance the software.

Proprietary Tools

Proprietary tools are software products that are created and controlled by an individual or organization. Users generally need to purchase licenses or subscriptions to access the software and its training materials. Unlike open-source software, the source code of proprietary tools is restricted and can only be modified by the owners or authorized developers.

As a result, users depend on the software provider for updates, bug fixes, and new features, which may sometimes involve additional costs. Although proprietary software may allow limited customization, the flexibility is usually not as extensive as with open-source tools. Common examples of proprietary security tools include Splunk SIEM solutions and Google SecOps.

Common Misconceptions About Open-Source Software

A frequent misconception is that open-source tools are less secure or less effective than proprietary alternatives. In reality, many open-source projects have become widely accepted industry standards over time.

Although attackers may attempt to exploit vulnerabilities in open-source software, the public availability of the source code often works as an advantage. Security professionals and developers worldwide can quickly inspect the code, identify issues, and release fixes. This collaborative approach can improve transparency and speed up vulnerability detection and resolution.

Examples of Open-Source Security Tools
Linux

Linux is a widely used open-source operating system that provides users with extensive control and customization options, mainly through its command-line interface (CLI). An operating system acts as the connection between the user and the computer’s hardware, while also managing software applications and system resources.

There are many Linux distributions available, each designed for different purposes such as servers, cybersecurity, development, or personal use. Linux is especially popular in cybersecurity because of its flexibility and powerful command-line tools.
 
Suricata

Suricata is an open-source tool used for network monitoring, traffic analysis, and threat detection. It examines network activity to identify suspicious behavior and creates logs that help security teams investigate possible threats, risks, or vulnerabilities.

The software was developed by the Open Information Security Foundation (OISF), an organization focused on maintaining Suricata as a free and publicly accessible project. Suricata is commonly used in both private companies and public organizations, and it can integrate with SIEM platforms and other cybersecurity tools to strengthen network security monitoring.

These platforms enable organizations to collect, monitor, search, and analyze log data from multiple systems through interactive dashboards. By centralizing security information, Splunk helps security teams gain visibility into organizational activities and manage security operations more effectively.

Below are some important Splunk dashboards and their functions:

Security Posture Dashboard

The security posture dashboard is mainly used in Security Operations Centers (SOCs). It presents an overview of significant security events and trends from the past 24 hours. Analysts use this dashboard to confirm whether security controls and policies are functioning properly. It also supports real-time monitoring of suspicious activities, such as unusual traffic coming from a particular IP address.

Executive Summary Dashboard

The executive summary dashboard provides a broad overview of an organization’s security condition over a period of time. It helps security teams evaluate security performance and identify areas that need improvement. Analysts often use this dashboard to present summarized reports of incidents, trends, and risks to management or stakeholders.

Incident Review Dashboard

The incident review dashboard helps analysts investigate suspicious events by displaying patterns related to security incidents. It prioritizes high-risk activities that require immediate attention and often includes a visual timeline of events leading up to an incident, making investigations more efficient.

Risk Analysis Dashboard

The risk analysis dashboard focuses on measuring the risk associated with different assets such as users, devices, or IP addresses. It tracks unusual behaviors, including abnormal login times or unexpected increases in network activity. Analysts use this information to identify vulnerabilities and prioritize security responses based on risk levels.

Google SecOps (Chronicle)

Chronicle is a cloud-native SIEM platform developed by Google. It is designed to store, search, and analyze large amounts of log data to detect potential cybersecurity threats, risks, and vulnerabilities. Chronicle allows analysts to investigate security events related to specific users, assets, domains, or IP addresses.

The platform includes several dashboards that assist security professionals in monitoring and analyzing organizational activity.

Enterprise Insights Dashboard

The enterprise insights dashboard displays recent alerts and identifies suspicious domain names, also known as Indicators of Compromise (IOCs). Each alert is assigned a confidence score and severity level to indicate the probability and impact of a threat. Analysts use this dashboard to detect suspicious access attempts or unusual activities affecting important systems or applications.

Data Ingestion and Health Dashboard

This dashboard monitors the flow and processing of log data into Chronicle. It displays information such as the number of event logs received, log source availability, and processing success rates. Security teams use it to verify that logging systems are functioning correctly and that important security data is not being lost.

IOC Matches Dashboard

The IOC matches dashboard highlights major threats, vulnerabilities, and risks detected within the organization. It helps analysts observe trends involving suspicious IP addresses, domains, or devices over time. This information supports threat prioritization and further investigation into unusual activity.

Main Dashboard

The main dashboard offers a high-level summary of security events, alerts, and data ingestion activities across the organization. Analysts can use it to monitor trends such as increases in failed login attempts, suspicious network behavior, or abnormal activity across multiple systems and locations.

Rule Detections Dashboard

The rule detections dashboard provides statistical information about recurring alerts and incidents generated by detection rules. Analysts use this dashboard to identify commonly triggered threats and evaluate the effectiveness of security rules. This helps organizations improve mitigation strategies and reduce future risks.

User Sign-In Overview Dashboard

The user sign-in overview dashboard focuses on user authentication behavior throughout the organization. It allows analysts to monitor login activities and identify suspicious patterns, such as simultaneous logins from different geographic locations. This information helps security teams detect compromised accounts and strengthen account security measures.
